WPA2/WPA3 具体实现 + 四帧握手逐帧拆解

第 9 课讲了 4-Way Handshake 的设计框架——四个 EAPOL-Key 帧派生 PTK、下发 GTK。这课是它的实现层深入：WPA2 和 WPA3 各自怎么把 PMK 拿到手，然后把四个握手帧逐帧摊开，看每个帧里到底装了什么字段、谁在什么时候算什么、Wireshark 怎么读。

这是抓包级、代码级的细节课。学完你应该能在 Wireshark 里一眼读懂 M1-M4，并在代码里定位"PMK 派生在哪、PTK 安装在哪、MIC 校验失败报什么错"。

支点：WPA2 vs WPA3，差异全在"PMK 怎么来"

本课支点 / Core Insight

第 9 课讲过：4-Way Handshake 本身是固定的，无论 WPA2 还是 WPA3 都要走 M1-M4。两代的全部差异都在握手之前怎么把 PMK 准备好：

  • WPA2-PSK：PMK = PBKDF2(密码, SSID) —— 双方各自离线算，无网络交互
  • WPA3-SAE：PMK 由 Dragonfly 握手（Commit/Confirm 两对帧）在线协商，与密码非线性关联

PMK 准备好之后，两条路汇合，走完全相同的 4-Way Handshake。所以这课分两大块：先讲两种 PMK 派生，再统一讲四帧握手。

① WPA2-PSK 的 PMK：PBKDF2 一步派生

WPA2-Personal（家用）最简单。密码（passphrase）和 SSID 在本地就能算出 PMK，不需要任何网络交互：

PMK = PBKDF2-HMAC-SHA1(passphrase, SSID, 4096, 32) │ │ │ │ │ │ │ │ │ └─ 输出 32 字节 (256 bit) │ │ │ └─ 4096 轮迭代 │ │ └─ SSID 作盐 │ └─ HMAC-SHA1 作伪随机函数 └─ 密码 (8-63 字符 ASCII)

四个关键参数，每个都有设计意图：

  • 密码 8–63 字符：WFA 认证要求 ≥8 字符。短密码是 WPA2 的最大软肋。
  • SSID 当盐：这是反预计算的关键。同一密码"TP-LINK_1234"和"Netgear_5678"派生出完全不同的 PMK。所以早期有人用常见 SSID（"linksys"、"netgear"）做预计算彩虹表，但 SSID 一变就失效。
  • 4096 轮：故意慢。单次 PBKDF2 在 CPU 上要几十毫秒，抬高离线爆破成本。但 GPU/ASIC 并行后仍可每秒试几万到几十万个密码——这就是为什么弱密码的 WPA2 网络几小时就能被 aircrack-ng 破。
  • 32 字节输出：正好是 PMK 的长度。HMAC-SHA1 单次输出 20 字节，要 32 字节需要 PBKDF2 内部跑两个块拼接。

派生完后，STA 和 AP 都持有同一个 PMK，但 PMK 从不在空中传。接下来直接进 4-Way Handshake。

桥梁 / From BT

BLE 的 IRK（Identity Resolving Key）也是从长期密钥派生，思路类似：根密钥本地持有，派生过程不联网。但 BLE 的 TK（Temporary Key）派生用的是 AES-CMAC，不是 PBKDF2。功能对应，算法不同。

② WPA3-SAE 的 PMK：Dragonfly 两阶段在线协商

WPA3-Personal 用 SAE (Simultaneous Authentication of Equals) 替代 PSK。SAE 基于 Dragonfly 握手（RFC 7664，PAKE 协议家族），核心改动：PMK 不再等于 f(密码)，而是经过椭圆曲线上的随机运算派生，抓包无法离线爆破。

SAE 分两个阶段，都发生在关联之前（取代了 WPA2 里 PSK 那个"无网络交互"的环节）：

阶段 A: 双方各自派生 PWE (Password Element) STA: 密码 + MAC ──H2E 映射──▶ PWE (曲线上一个秘密点) AP: 密码 + MAC ──H2E 映射──▶ PWE PWE 派生方法: H2E (新, 推荐) 或 HNP (老, 有侧信道漏洞) PWE 是后续所有运算的基础, 不在空中传 阶段 B: Commit / Confirm 两对帧在线协商 PMK STA (手机) AP (路由器) │ │ │ SAE Commit │ │ ──────── Commit (scalar-i, element-i) ─▶│ STA 发它的标量和元素 │ ◀──────── Commit (scalar-r, element-r) ── │ AP 回它的标量和元素 │ │ │ 双方各自用 PWE + 对方的 element + 自己的 │ │ 随机私钥, 算出共享秘密 → 派生 PMK │ │ │ │ SAE Confirm │ │ ──────── Confirm (token-i) ────────────▶│ STA 发确认 token (用 PMK 签的) │ ◀──────── Confirm (token-r) ──────────── │ AP 回确认 token │ │ │ 双方验证对方 token = 确实算出了同一个 PMK ✓ │ │ │ │ ═══ 现在双方都有 PMK, 进关联+4-Way Handshake ═══

四个要点解释为什么 SAE 抗爆破：

  • PWE 派生不泄漏密码：H2E（Hash-to-Element）把密码确定性映射到曲线上一点，输出看不出密码信息。老方法 HNP（Hunting-and-Pecking）迭代试值，迭代次数泄漏密码信息——Dragonblood 侧信道攻击的基础。新固件应支持 H2E。
  • Commit 交换不泄漏 PMK：双方发的 scalar/element 是 PWE + 随机私钥的运算结果。攻击者抓到 Commit 也推不出 PMK——因为这等价于解椭圆曲线离散对数（ECDLP，目前不可行）。
  • Confirm 防中间人：双方互发用 PMK 签的 token，验证对方确实算出了同一个 PMK。攻击者不知道密码 → 算不出 PWE → 算不出正确 PMK → token 校验失败。
  • 在线才能验证猜测：要试一个密码，必须和 AP 真做一次 SAE 交换（在线）。AP 可以限速/锁定，离线爆破不可行。这是 SAE 相对 PSK 的根本优势。
SAE 的代价

椭圆曲线运算（标量乘、点加）比 PBKDF2 的 HMAC 重得多，SAE 对 STA 和 AP 的 CPU 都有要求。低端 IoT 芯片做 SAE 可能要几百毫秒。这也是为什么 WPA3 强制 PMF 但允许 WPA3 Transition Mode（WPA2/WPA3 混合）——给老设备留过渡。

⚠️ 常见误区：既然都走四次握手，PMK 不是还能算出来？

这是学完前两节后最自然的疑问："4-Way Handshake 在 WPA2 和 WPA3 里完全一样，那 PMK 不还是能算出来？这次会话的数据不还是能被解密？"

结论先给：WPA2 里能，WPA3 里不能。差别在 PMK 的派生方式，不在四次握手。

误区根源：PMK 是不是"可被旁观者复算"

两种 PMK 派生公式对比：

WPA2-PSK: PMK = PBKDF2(密码, SSID, 4096) ↑ 完全确定, 抓包者能离线复算 WPA3-SAE: PMK = f( 密码派生的PWE, STA的随机私钥, AP的随机私钥 ) ↑ ↑ ↑ 可复算 一次性随机 一次性随机 用完即弃 用完即弃 不在空中传 不在空中传

WPA3 的 PMK 依赖两个一次性随机私钥，它们用完即弃、从不在空中明文传。攻击者抓到的 SAE Commit 帧里只有它们的"运算结果"（scalar/element），从结果反推随机私钥等价于解椭圆曲线离散对数问题（ECDLP）——目前公认不可行的数学难题。

攻击者能力能解密这次会话吗原因
只抓包（被动监听） ❌ 不能 算不出 PMK → 算不出 PTK → 解不了 CCMP
抓包 + 事后知道密码 ❌ 不能（前向保密） 当时的一次性随机私钥已销毁，事后知道密码也复算不出当时的 PMK
抓包 + 当场攻破设备内存 ✅ 能 实时入侵，门槛远高于被动抓包

中间那一行就是前向保密 (Forward Secrecy)：密码今天泄露，解密不了昨天抓的包。WPA2-PSK 没有这个性质（密码泄露 → 所有历史抓包全解）。

攻击者要验证一个密码猜测，唯一办法是和 AP 真做一次在线 SAE 交换（AP 生成新的随机私钥参与运算）。在线 = 可被限速、锁定、检测。这就是 SAE 根治离线爆破的原因——不是改了四次握手，而是让 PMK 从"密码的确定性函数"变成了"一次性随机协商的产物"。

类比你熟悉的：这就是 Diffie-Hellman 密钥交换的思路——双方用各自的随机私钥在公开信道协商共享秘密，旁观者算不出来。SAE = PAKE（密码认证的 DH），BLE 配对里用的 ECDH 也是同源思想。

③ 4-Way Handshake：WPA2 和 WPA3 的汇合点

无论 PMK 来自 PBKDF2（WPA2）还是 SAE（WPA3），从这一步开始两条路完全一样。四帧握手用同一个 PMK 派生 PTK。下面把四帧逐个拆到字段级：

STA AP │ │ │ PMK 已就绪 (WPA2: PBKDF2 算出; WPA3: SAE 算出) │ │ │ │ M1 (AP→STA, 发 ANonce) │ │ ◀─────── EAPOL-Key ──────────────│ │ Key Info: ACK=1 (要 STA 回复) │ │ ANonce (AP 的随机数) │ │ Replay Counter = N │ │ │ │ STA 生成 SNonce, 现在能算 PTK: │ │ PTK = PRF(PMK, ANonce, SNonce, MAC_AP, MAC_STA) │ │ │ M2 (STA→AP, 发 SNonce + MIC) │ │ ──────── EAPOL-Key ──────────────▶│ │ Key Info: MIC=1 │ │ SNonce (STA 的随机数) │ │ MIC (用 KCK 算, 证明 PTK 正确) │ │ RSN-IE (STA 的安全能力) │ │ Replay Counter = N (回显) │ │ │ │ AP 算出 PTK│ │ AP 验 MIC = STA 算对了 ✓│ │ AP 检查 RSN-IE 防降级│ │ │ │ M3 (AP→STA, 发 GTK + RSN-IE + MIC) │ │ ◀─────── EAPOL-Key ──────────────│ │ Key Info: ACK=1, MIC=1, Install=1, Encrypted=1, Secure=1 │ MIC (AP 也证明算对了) │ │ Key Data: GTK (用 KEK 加密) + AP 的 RSN-IE │ Replay Counter = N+1 │ │ AP install PTK│ │ │ │ STA 验 MIC, install PTK + GTK │ │ │ │ M4 (STA→AP, 纯确认) │ │ ──────── EAPOL-Key ──────────────▶│ │ Key Info: MIC=1, Secure=1 │ │ MIC (确认安装完毕) │ │ Replay Counter = N+1 (回显) │ │ │ │ ═══ 双方已授权, 开始加密数据帧 ═══ │

关键：M4 之后 Key Information 里的 Secure bit 置 1，表示握手完成、后续进入"安全态"。这是状态机的分水岭。

怎么记这张表

三条规律：

  1. Nonce 各发一次：ANonce 只在 M1，SNonce 只在 M2。凑齐了双方都能算 PTK。
  2. MIC 从 M2 开始：M1 还没 PTK 签不出 MIC；M2-M4 都有。
  3. M3 是"大包"：GTK、Install、Secure、Encrypted 都在 M3 出现——M3 是信息量最大的一帧。

记不住细节没关系，抓包时对照这张表读一遍就熟了。

WPA2 vs WPA3 完整流程对比

把两种实现的完整时间线放一起对比，差异和共性一目了然：

阶段 WPA2-Personal (PSK) WPA3-Personal (SAE)
PMK 准备 双方各自 PBKDF2(密码, SSID, 4096)，无网络交互 SAE Commit/Confirm 两对帧在线协商（关联之前）
扫描/认证/关联 Beacon/Probe → Auth → Assoc（第 7 课） Beacon/Probe → SAE Commit/Confirm → Auth → Assoc
4-Way Handshake M1-M4（用 PSK 派生的 PMK） M1-M4（用 SAE 派生的 PMK）—— 完全相同
PMF (管理帧保护) 可选（通常关） 强制
加密算法 CCMP（AES） CCMP（AES）+ 可选 GCMP（WPA3 进阶）
密码要求 ≥8 字符 ≥12 字符（WFA 认证要求）
抗离线爆破 ❌ 抓 4-Way 可爆破 ✅ 抗爆破
前向保密 ❌ 无 ✅ 有（每会话 PMK 新鲜）
关键认知

WPA3 没有重新发明 4-Way Handshake，也没有换加密算法（仍用 AES-CCMP）。它做的只有两件事：①把 PMK 的来源从"离线 PBKDF2"换成"在线 SAE"，根治离线爆破；②强制 PMF，根治 Deauth 攻击。4-Way Handshake 和 CCMP 是 WPA2/WPA3 共享的基石，你做固件时这两块的代码 WPA2/WPA3 通用。

对企业版（802.1X）的一句话补充

WPA2/WPA3-Enterprise 走 802.1X/EAP：STA 和 RADIUS 服务器做 EAP 认证（EAP-TLS/PEAP/TTLS），AP 当中间人，认证成功后 RADIUS 把 MSK 给 AP，AP 和 STA 各自从 MSK 截取出 PMK。之后同样走标准的 4-Way Handshake。所以对企业版，PMK 来自 EAP/MSK；对 Personal 版，PMK 来自 PSK 或 SAE——4-Way Handshake 是三条路的共同汇合点。

对驱动栈：代码里这些在哪

对驱动重要 / For Driver Stack

WPA2/WPA3 实现在 Linux 栈里的代码分布：

  • PMK 派生（PBKDF2 / SAE）：全在 wpa_supplicant 用户态。内核和驱动看不到 PMK 的派生过程，只收到 install 命令。SAE 的椭圆曲线运算（PWE/Commit/Confirm）也在 wpa_supplicant，靠 crypto.h 抽象的 ECC 库实现。
  • 4-Way Handshake 状态机：wpa_supplicant 的 wpa.c（wpa_sm 结构）维护 M1-M4 状态，处理收到的 EAPOL-Key 帧、派生 PTK、校验 MIC、生成回复。
  • 密钥 install 到硬件：wpa_supplicant 算出 PTK/GTK 后，通过 nl80211 的 NL80211_CMD_NEW_KEY 把密钥下发到内核 cfg80211 → mac80211 → 驱动 → 芯片 key table。芯片硬件做 CCMP 加解密。
  • SAE 新特性：内核侧要支持 NL80211_CMD_EXTERNAL_AUTH（SAE 在用户态做，内核触发）。驱动需在 Assoc Request 里正确设置 SAE 认证类型。老的只支持 WPA2 的固件不能直接上 WPA3，要确认芯片固件和 wpa_supplicant 版本。
  • 调试：wpa_supplicant -ddd 日志能看到完整的 M1-M4 交换、PTK 派生、MIC 校验。MIC 校验失败（MIC check failed）通常是 PMK 派生错误（密码/SSID 拼接错）或 KCK 截取错误。

自检：实现细节吃透了吗

这课的钉子：差异在 PMK 来源，汇合在 4-Way

学完这课，你脑子里应该有两条清晰的路：

  1. WPA2-PSK：密码 + SSID → PBKDF2(4096轮) → PMK（离线，可爆破）→ 4-Way
  2. WPA3-SAE：密码 → PWE(H2E) → Commit/Confirm 在线协商 → PMK（抗爆破）→ 4-Way

两条路在 4-Way Handshake 汇合，M1-M4 完全相同。4-Way 的四帧各有分工：M1 发 ANonce、M2 发 SNonce+MIC、M3 发 GTK+Install、M4 确认。Key Information 字段是读 EAPOL-Key 帧的入口。

你现在能做的：

  • 在 Wireshark 里逐字段读懂 EAPOL-Key M1-M4
  • 说清 WPA2 和 WPA3 在哪一步开始相同、哪一步不同
  • 理解 PBKDF2 的 SSID 盐、4096 轮、SAE 的 PWE/Commit/Confirm 各自的设计意图
  • 定位 MIC 校验失败、4-Way 超时该查 wpa_supplicant 的哪段代码
  • 知道为什么做新固件该上 WPA3 + H2E + PMF
