STA 关联流程 —— 协议栈移植的主战场

这是你做 Wi-Fi 驱动移植最核心的一课。前面六课讲的（RF、OFDM、CSMA/CA、帧结构）都是"基础设施"，这一课讲的是状态机 —— 一个 STA 怎么从"什么都不行"走到"能传数据"。这正是你做 BT 协议栈移植时的老本行，只是把 BLE 的 Standby → Advertising → Connection 换成 Wi-Fi 的版本。

第 6 课讲了管理帧是状态机的载体。这课就把状态机本身摊开给你看：哪些状态、什么帧触发迁移、迁移失败会怎样。关联连不上、漫游断流、连上不能传 —— 90% 的 Wi-Fi 协议栈 bug 都在这条流程上。

支点：802.11 的三态状态机

本课支点 / Core Insight

802.11 把一个 STA 的生命周期划成三个状态，状态之间靠管理帧迁移。每个状态只允许发对应"等级"的帧（Class 1/2/3）。这是协议栈状态机的官方骨架，比 BLE Link Layer 的状态机更"扁平"但更细：

  • State 1 · 未认证 (Unauthenticated)：刚开机，几乎什么都干不了，只能扫描
  • State 2 · 已认证未关联 (Authenticated, Unassociated)：过了 802.11 认证，还没加入 BSS
  • State 3 · 已关联 (Associated)：正式加入 BSS，可以发数据帧（WPA 下还要 4-Way Handshake 才真正能传）

状态机和帧等级的关系：

Authentication State 1 ──────────────────▶ State 2 未认证 (Unauth) ◀────────────────── 已认证未关联 Deauthentication (Auth, Unassoc) (任意状态打回) │ │ Association Disassociate ▼ ◀────────────────── State 3 (保留认证) 已关联 (Associated) │ + 4-Way Handshake → "已授权" ▼ Class 1 帧: Probe, Beacon, Auth → State 1 起允许 Class 2 帧: Association Req/Resp → State 2 起允许 Class 3 帧: 数据帧, Deauth, Disassoc → State 3 起允许
桥梁 / From BT

对照 BLE Link Layer：Standby → Advertising → Initiating → Connection。Wi-Fi 没有 BLE 的"广播态"概念（AP 永远在广播 Beacon，STA 不广播），但状态机的思想完全一致 —— 每个状态能做什么、哪些 PDU 触发迁移、迁移失败如何回退，你都做过。这是你迁移 BT 协议栈经验最直接的落点。

完整序列：从开机到能传数据

下面是开放式 (Open) + WPA2-PSK 网络下，STA 从无到有连上 AP 的完整管理帧序列。先看全貌，再逐段拆：

STA (手机) AP (路由器) │ │ │ ① 扫描 / Scanning │ │ ◀──────────── Beacon ────────────────── │ (被动: AP 周期广播) │ ──────────── Probe Request ────────────▶ │ (主动: STA 问"有人在吗") │ ◀──────────── Probe Response ─────────── │ │ │ │ ② 认证 / Authentication (注意: 不是安全认证!) │ │ ──────────── Auth Seq 1 ────────────────▶ │ algorithm=0 (Open System) │ ◀──────────── Auth Seq 2 ──────────────── │ Status=0 (成功) │ → State 2 │ │ │ │ ③ 关联 / Association │ │ ──────────── Association Request ────────▶│ 能力 + 速率 + 监听信道 │ ◀──────────── Association Response ─────── │ Status=0 + AID=42 │ → State 3 │ │ │ │ ④ 授权 / Authorization (WPA 才需要) │ │ ──────────── EAPOL-Key 1/4 ──────────────▶ │ │ ◀──────────── EAPOL-Key 2/4 ─────────────── │ 4-Way Handshake │ ──────────── EAPOL-Key 3/4 ──────────────▶ │ 协商 PTK（加密密钥） │ ◀──────────── EAPOL-Key 4/4 ─────────────── │ │ → 已授权, 可传数据│ │ │ │ ⑤ 数据传输 / Data │ │ ═══════════ 加密数据帧 (Protected=1) ═══▶│ │ ◀═══════════ 加密数据帧 ════════════════ │

注意这五段里的两个"伪命题"，这是新人最常踩的坑：

  • 步骤②不验证身份：802.11 Authentication 几乎永远成功，它不是安全认证，只是状态机过场。真正的身份认证在步骤④。
  • 步骤③不等于能传数据：WPA 网络下，Association 成功只到 State 3，还要过 4-Way Handshake 进"已授权"才能上行。Open 网络下 State 3 就能传。

① 扫描：怎么发现 AP

STA 要先知道周围有哪些 AP，才能挑一个连。两种扫描方式：

方式怎么做时延耗电典型场景
被动扫描 (Passive) STA 在每个信道安静听，等 AP 周期广播的 Beacon（典型 102.4 ms 一次） 慢（每个信道要等 ~100 ms） 低 隐藏 SSID 网络、法规限制主动扫描的地区
主动扫描 (Active) STA 在每个信道发 Probe Request，匹配的 AP 回 Probe Response 快 略高（要发帧） 绝大多数手机/笔记本的默认行为

扫描要在多个信道之间跳（2.4 GHz 有 1–13，5 GHz 几十个），每信道停一下听/问，这就是为什么完整扫描要花 1–2 秒。Beacon 和 Probe Response 内容几乎一样：SSID、BSSID、Capability Info（能力位）、支持的速率、加密套件、TIM 等。STA 拿这些信息挑 AP（一般按信号强度 RSSI 排序）。

对驱动重要 / For Driver Stack

扫描是 mac80211 / cfg80211 里最复杂的模块之一。上层（wpa_supplicant / Android WiFi HAL）发 NL80211_CMD_TRIGGER_SCAN 命令，驱动负责切信道 → 收集 Beacon/Probe → 上报 BSS 列表。每个信道停留时间（dwell time）、跳信道时怎么保证不漏帧、扫完怎么回到原信道——这些都是你移植时要调的参数。扫描慢、扫不到 AP，是固件最常见的 bug 之一。

② 认证：802.11 的"假"认证

选好 AP 后，STA 进入 State 1 → State 2 的迁移：802.11 Authentication。这是新概念里最容易误解的一个，先把名字的坑填平：

⚠️ 名字陷阱

802.11 标准里的 "Authentication (认证)" 不等于 安全认证。它是一个固定两帧的握手，Open System 模式下基本永远成功，作用只是"告诉 AP：我要进入 State 2"。真正验证身份（PSK 密码对不对、802.1X 证书）发生在关联之后的 4-Way Handshake / EAP 流程。

标准这么命名是历史包袱：1997 年的 802.11 原版里这一步还能做 WEP Shared Key 认证（带挑战-应答），后来 WEP 被淘汰，Open System 成了唯一在用的算法，这步就退化为空壳，但名字和帧格式留了下来。

Open System Authentication 的两帧：

帧方向关键字段
Authentication (Seq 1) STA → AP Authentication Algorithm = 0 (Open System)；Transaction Seq = 1
Authentication (Seq 2) AP → STA Algorithm = 0；Seq = 2；Status Code = 0 (Success)

AP 在 State 1 时收到 Auth Request，正常情况下没有理由拒绝（除非 ACL 黑名单、STA 太多）。回个 Auth Seq 2，状态机推进到 State 2。

抓包看 802.11 Auth 帧时，Auth Seq 1/2 的内容基本是空的——Open System 不带任何挑战/应答数据。如果你看到带 challenge text 的 Auth 帧，那是 90 年代的 WEP Shared Key 模式，现在几乎绝迹了。

③ 关联：真正加入 BSS

State 2 → State 3 的迁移靠 Association (关联)。这一步是"实质性"的——AP 在这里给 STA 分配资源、协商能力、分配 AID。

帧方向关键字段
Association Request STA → AP Capability Info（我要的能力）、Listen Interval（省电相关）、Supported Rates、SSID
Association Response AP → STA Status Code、AID（1–2007）、Supported Rates、协商后的能力

关联请求里 STA 把自己能干的事（Capability Info 位域：支持哪些速率、要不要 QoS、要不要短前导等）告诉 AP，AP 在 Response 里回"哪些同意、哪些不行"，并分配一个 AID (Association ID, 关联标识符)，范围 1–2007。

AID 是什么，为什么重要

AID 是 AP 给每个关联 STA 的唯一编号。它的作用到省电课才显出来：

  • PS-Poll 帧的 Duration/ID 字段存 AID，STA 用它向 AP"取快递"（缓存的下行帧）。
  • TIM（在 Beacon 里）是一个位图，按 AID 标记哪个 STA 有缓存的下行帧。

AID 范围 1–2007 限制了一个 BSS 最多 2007 个关联 STA——这是 802.11 单 AP 的硬上限。实际产品里因内存/调度限制远低于此。

AP 为什么会拒绝关联？常见 Reason Code：能力不匹配（STA 要 QoS 但 AP 不支持）、STA 数已满、ACL 黑名单、RSSI 太低。你做协议栈时这些失败原因都要正确上报给上层。

④ 授权：4-Way Handshake（WPA 才有）

这是安全课的内容，这里只点一下它在状态机里的位置。Association 成功 → State 3，但 WPA/WPA2/WPA3 网络下，State 3 还不能立刻发用户数据。STA 和 AP 要先做 4-Way Handshake（四个 EAPOL-Key 帧来回），协商出单播加密密钥 PTK (Pairwise Transient Key) 和组播密钥 GTK。握手完成后 STA 进入"已授权"状态，才能发加密的 Protected=1 数据帧。

为什么状态机要分这么细

把"关联"和"授权"分开，让 802.11 MAC 层和安全层解耦：同一个状态机骨架能挂 Open / WPA2-PSK / WPA2-Enterprise (802.1X) / WPA3 各种安全方案。Open 网络 State 3 直接传明文；WPA 网络 State 3 后多走 EAPOL；Enterprise 还要在 4-Way 之前先走完整的 EAP/RADIUS。这是协议分层的经典设计，也是你做协议栈移植要尊重的边界——mac80211 管 State 1→2→3，wpa_supplicant 管 4-Way Handshake。

断开：Deauth vs Disassociate

连接的对称面是断开。有两个帧，区别在回退到哪个状态：

帧状态回退到语义触发场景
Disassociate State 3 → State 2 取消关联，保留认证 AP 踢人、STA 主动下线（不关 Wi-Fi）
Deauthentication 任意 → State 1 取消认证，彻底打回原形 关 Wi-Fi、超时、严重错误、攻击（Deauth 攻击）

两个帧都带 Reason Code（为什么断），这是抓包定位断连原因的钥匙。常见的有：Reason 7（Class 3 帧来自未关联 STA）、Reason 15（4-Way Handshake 超时）、Reason 23（802.1X 认证失败）。

漫游：Reassociation 的角色

STA 在同一个 ESS 内（多 AP 共用一个 SSID）从原 AP 走到新 AP 覆盖范围时，要做漫游 (Roaming)。这时候不是从零开始的 Association，而是 Reassociation (重关联)：

  • STA 已经在 State 3（和新 AP 没关联，但和原 AP 关联着）。
  • STA 向新 AP 发 Reassociation Request——比 Association Request 多一个字段 "Current AP"（原 AP 的 MAC）。
  • 新 AP 回 Reassociation Response（带新 AID）。
  • 新 AP（在同一个 DS 内）通过有线网通知原 AP："这个 STA 转给我了"，原 AP 把缓存的帧转给新 AP。

漫游是 Wi-Fi 协议栈最难调的场景之一——切换太晚掉线、切换太早丢包、4-Way Handshake 重做耗时长、PMK 缓存 (802.11r) 没生效。这是你以后做固件的主战场之一。

桥梁 / From BT

BT 没有完全对应的概念，但 BLE 的连接参数更新和PHY 更新流程在精神上接近——都是连接已经存在时的"重新协商"。Wi-Fi 漫游更接近蜂窝的切换 (handover)：从一个接入点转到另一个，状态要迁移、上下文要搬。你做 sub-GHz 协议栈时如果做过中继/接力，这种"换锚点"的思路也通用。

对驱动栈：这套流程落在代码哪里

对驱动重要 / For Driver Stack

这课内容在 Linux/mac80211 栈里的对应点：

  • 状态机本体：mac80211 维护 ieee80211_if_managed 子状态（扫描中/已认证/已关联/关联中），对应 802.11 的 State 1/2/3。你移植时这个状态机要么用 mac80211 的实现，要么自己重写一份——这是 BT 协议栈状态机经验最直接迁移的地方。
  • 管理帧收发：ieee80211_sta_tx_auth/assoc 这组函数构造 Auth/Assoc Request；rx 路径按 subtype 分发到 ieee80211_rx_mgmt_auth/assoc。帧字段填充错了（Capability Info 漏位、AID 解析错）直接导致关联失败。
  • 跨进程边界：mac80211 管到 State 3；wpa_supplicant（用户态）接管 4-Way Handshake 和漫游决策。两边的接口是 nl80211（netlink），调试时要同时看内核日志和 wpa_supplicant 日志。
  • 调试黄金组合：wpa_supplicant -ddd（看状态机日志）+ monitor mode 抓包（看真实帧）+ dmesg（看驱动日志）。关联失败时三路日志对时间戳，问题在哪一段一眼看出。
  • 常见 bug：①扫描 dwell time 太短漏 AP；②Auth/Assoc 帧地址填错（参考第 6 课 ToDS/FromDS）；③Capability Info 位填错导致 AP 拒关联；④4-Way Handshake 超时（密钥派生 bug）；⑤漫游时没刷旧密钥导致新 AP 拒收。
