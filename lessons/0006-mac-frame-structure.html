802.11 MAC 帧结构 —— 协议栈工程师的主场

这课完全在你的强项区。帧解析、字段含义、状态机 —— 这就是你做协议栈每天干的事。我会用协议栈视角讲，而不是管理员视角。你会看到 Wi-Fi 的 MAC 帧设计精妙在哪，以及它为什么比你做过的 BT 帧复杂。

上一课讲了 CSMA/CA 怎么抢到介质（谁来发、什么时候发）。这课讲 抢到之后发的"东西"长什么样 —— MAC 帧。这是你做驱动协议栈最常打交道的字节布局。

支点：三类帧，各司其职

本课支点 / Core Insight

802.11 把帧分成三类，这是理解整个协议的钥匙：

  • 管理帧 (Management)：建立/维护/断开连接（关联、扫描、认证）
  • 控制帧 (Control)：辅助数据帧传输（ACK、RTS、CTS），自己不承载数据
  • 数据帧 (Data)：承载上层数据（IP 包等）

对比 BT：BLE 的广告/PDU/Control PDU 也是按"管理/数据"分，但 802.11 的分法更彻底 —— 管理帧本身就是一组独立的状态机（关联、漫游、QoS 协商全靠它）。

帧整体布局：一张图记住

这是通用 MAC 帧的布局。注意Address 4 是可选的（只有 WDS 无线桥接才用）：

几个关键数字记住：

  • MAC 地址：每个 6 字节（48 bit）—— 和以太网一致
  • 固定头：24 字节（无 Addr4）或 30 字节（WDS 含 Addr4）
  • Body：可变，0–2312 字节（加密后最多 2346）
  • FCS：4 字节 CRC-32，校验失败丢弃且用 EIFS 退避（回看第 5 课）
桥梁 / From BT

BLE 的 Link Layer PDU 头是 16 bit（2 字节），极简。Wi-Fi MAC 头 24 字节起步，因为要塞 3-4 个 MAC 地址（6B×3-4=18-24B）。为什么这么多地址？下一节揭晓 —— 这是 Wi-Fi 帧设计的精髓。

四个地址字段：为什么这么多？

这是这课最该吃透的点。Wi-Fi 帧有四个地址字段，能扮演 5 个角色：

角色含义类比
DA Destination Address，最终目的 MAC收件人
SA Source Address，原始发送方 MAC寄件人
BSSID这个 BSS 的标识（AP 的 MAC）邮局代号
RA Receiver Address，本跳接收方中转站
TA Transmitter Address，本跳发送方当前快递员

为什么需要 5 个角色？因为 AP 是桥接节点 —— 帧从无线进入有线网时，"本跳接收方"(RA=AP) 和 "最终目的地"(DA=服务器) 是两个不同的地址。以太网帧只有 DA/SA 两个，Wi-Fi 必须多带信息表示"这一跳发给谁、最终要去哪"。

两条铁律

Addr1 永远是 RA（本跳接收方）—— 决定"谁该收这个帧"

Addr2 永远是 TA（本跳发送方）—— 决定"该回 ACK 给谁"

这两条恒成立，不管帧是什么类型。其余地址字段由 ToDS/FromDS 两个 bit 决定。

ToDS / FromDS：四地址映射的精髓

Frame Control 里有 ToDS 和 FromDS 两个 bit，表示帧是否要去往/来自分布式系统（DS，通常是有线网）。这两个 bit 决定了 Addr1-4 各自装什么。这张表是协议栈工程师必须背下来的：

ToDSFromDSAddr1 (RA)Addr2 (TA)Addr3Addr4场景
00 DASABSSID— 管理帧 / 同一 BSS 内通信
01 DABSSIDSA— AP → STA（下行，帧来自 DS）
10 BSSIDSADA— STA → AP（上行，帧要去 DS）
11 RATADASA WDS 无线桥接（AP 到 AP）

怎么记？三个场景对应三行：

① STA → AP（上行）：手机发数据给 AP。RA=AP（BSSID），TA=手机自己（SA）。最终目的地在有线网（DA）。
→ ToDS=1, FromDS=0，Addr1=BSSID, Addr2=SA, Addr3=DA。

② AP → STA（下行）：AP 把有线网来的数据发给手机。RA=手机（DA），TA=AP（BSSID），原始来源在有线网（SA）。
→ ToDS=0, FromDS=1，Addr1=DA, Addr2=BSSID, Addr3=SA。

③ 管理帧（ToDS=FromDS=0）：Beacon、Probe、Association 这些不走有线网，三个地址够用：Addr1=DA, Addr2=SA, Addr3=BSSID。

④ WDS（ToDS=FromDS=1）：无线桥接（AP 之间无线相连），两个方向都跨 DS，需要四个地址才能说清 RA/TA/DA/SA。这是唯一用满 Addr4 的场景。

对驱动重要 / For Driver Stack

这课对你做驱动最直接的落点：抓包分析。Wireshark + monitor mode 抓的 802.11 帧，你要能一眼看出"这是上/下行还是 WDS"。看 ToDS/FromDS 两个 bit 就行。调试关联失败、漫游异常时，这张表是解码钥匙 —— 比如 STA 发的 Association Request 永远是 ToDS=1, FromDS=0。

Frame Control：2 字节里的乾坤

帧头前 2 字节是 Frame Control，藏着所有"这个帧是什么"的信息。bit 布局：

bit: 0-1 2-3 4-7 8 9 10 11 12 13 14-15 │←Ver→│←Type→│←Subtype→│←ToDS→│←FromDS→│←MF→│←Retry→│←Pwr→│←MD→│←Prot→│←Ord→│ │ 2b │ 2b │ 4b │ 1b │ 1b │ 1b │ 1b │ 1b │ 1b │ 1b │ 1b │

关键位（做驱动最常打交道）：

  • Type (2b) + Subtype (4b)：帧类型和子类型。Type=00 管理、01 控制、10 数据。比如 Beacon = Type00/Subtype1000，ACK = Type01/Subtype1101。这是解析帧的第一步。
  • ToDS / FromDS (各 1b)：见上节，决定地址含义。
  • Retry (1b)：重传帧标志。接收方用它 + Seq Ctl 去重（同一个帧收到两次只处理一次）。
  • More Fragments (1b)：还有后续分片。配合 Seq Ctl 的 Fragment Number 重组。
  • Power Management (1b)：STA 告诉 AP"我要进省电模式"。AP 收到后为它缓存帧（配合 TIM 在 Beacon 里通知）。
  • Protected Frame (1b)：帧体已加密（WPA）。解析时先解密再看 Body。

Duration 与 NAV：CSMA/CA 的"虚拟侦听"

第 5 课讲 CSMA/CA 靠 CCA（物理侦听）判断信道忙闲。其实还有第二套机制 —— NAV（Network Allocation Vector，网络分配矢量），叫"虚拟载波侦听"。

原理：每个帧的 Duration 字段告诉所有听到的 STA「这组帧交换还要占信道多久」（μs）。听到这个值的 STA，把自己的本地计数器 NAV 设上，NAV 没归零就不尝试发送，哪怕物理 CCA 说信道闲。

物理侦听 + 虚拟侦听 = 双保险

STA 真正判断"能不能发"时，物理 CCA 和 NAV 同时检查：

  • CCA 说忙 → 不能发
  • NAV 说忙（别人声明要占着）→ 也不能发
  • 两个都闲 → 才进入 DIFS + backoff 流程

NAV 的价值：不用真的去听，光读 Duration 就知道信道被占了。这对隐藏节点尤其有用 —— 听不到数据帧本身，但能听到前置的 RTS/CTS 里的 Duration。

这也解释了为什么第 5 课的 RTS/CTS 能缓解隐藏节点：C 发不出 RTS 给 AP，但能听到 AP 回的 CTS 里的 Duration → 设置 NAV → 乖乖等。虚拟侦听绕过了"听不到"的物理限制。

三类帧速览：每个类型你最该知道的

管理帧（Management）—— 协议状态机的载体

帧方向用途
Beacon AP → 广播 周期广播 BSS 信息（SSID、能力、TIM）。STA 扫描靠它
Probe Request/ResponseSTA ↔ AP主动扫描。STA 发 Probe，AP 回 Response
Authentication STA ↔ AP 认证（开系统/WPA 握手的第一步）
Association Req/RespSTA ↔ AP关联（加入 BSS）。成功后 STA 才能发数据
Deauthentication / Disassociate双向断开连接

控制帧（Control）—— 不带数据，只辅助

帧用途
ACK 确认收到。仅 14 字节，最小帧
RTS / CTS隐藏节点缓解（第 5 课）。带 Duration 设 NAV
PS-Poll 省电模式 STA 向 AP 取缓存帧。Duration 字段存 AID
Block ACK Request / Block ACK批量确认（A-MPDU 聚合时用，提升效率）

数据帧（Data）—— 承载上层

最普通的是 Data subtype。QoS 数据帧（QoS Data）多 2 字节 QoS Control 字段，带 TID（接入类别）。你做驱动时 tx/rx 队列按 TID 分流，对应第 5 课讲的 EDCA 四个 AC。

对驱动栈：这套结构落在代码哪里

对驱动重要 / For Driver Stack

这课内容在你的日常工作的对应点：

  • rx 路径：固件把收到的帧 DMA 上来，驱动按 Frame Control 解析 Type/Subtype，分发给不同 handler（ieee80211_rx_mgmt 处理管理帧、ieee80211_rx_data 处理数据帧）。
  • 关联状态机：Beacon/Probe/Auth/Assoc 这组管理帧就是 STA 关联状态机的输入。你做 BT 协议栈时的状态机设计经验直接迁移。
  • 抓包调试：monitor mode + Wireshark 看原始 802.11 帧，这课的字段就是解码表。关联失败时看 Auth/Assoc 的来回，一眼定位问题。
  • NAV 处理：大部分在固件，驱动层看到的是"TX 被延迟"——延迟里可能含 NAV 占用。
  • 地址映射：STA 模式发上行帧时填 ToDS=1, Addr1=BSSID。AP 模式发下行填 FromDS=1。你移植驱动时这套填充逻辑常出 bug。
