WPA 安全 / 4-Way Handshake —— 兑现第 7 课的伏笔④

第 7 课那张连接序列图里画了 5 步，第 4 步 4-Way Handshake 我向你承诺了三次，现在兑现。这是 STA 从"关联成功但不能传数据"走到"真正能加密通信"的最后一公里，也是你做协议栈移植里和密钥管理打交道最多的代码路径。

这课的难点不在"四个帧的顺序"——那背一遍就记住了。难点在为什么要这么设计：为什么密钥不直接传？为什么要搞这么多层密钥？为什么 WPA2 能被破解、WPA3 又怎么修？理解了设计动机，整套机制就活了。

支点：WPA 安全解决三个问题

本课支点 / Core Insight

Wi-Fi 是开放广播介质——任何人都能听、都能发。WPA 安全要同时解决三个问题：

  1. 加密 (Confidentiality)：别人抓到空中帧也看不懂内容
  2. 完整性 (Integrity)：别人篡改帧能被发现（MIC）
  3. 身份认证 (Authentication)：确认对方确实知道密码

整个 WPA 体系（PSK、4-Way Handshake、PTK/GTK、CCMP）都是围绕这三个目标设计的。每个机制都能对应到"防什么攻击"。

先把802.11 Authentication ≠ 安全认证这个第 7 课的点再钉一遍：真正的安全认证发生在关联之后。家庭网络靠 PSK（双方预先从密码派生出 PMK），企业网络靠 802.1X/EAP（RADIUS 服务器认证），无论哪种，最后都要走 4-Way Handshake 把会话密钥派生出来。这课就讲这条主线。

密钥层级：从密码到数据帧密钥的四级金字塔

这是这课最容易晕、但最该吃透的部分。Wi-Fi 的密钥不是一把，是一整套层级，从"根"一路派生到"叶子"。先看全貌：

密码 (passphrase) │ │ PSK 模式: PBKDF2(passphrase, SSID, 4096轮) │ 802.1X 模式: RADIUS EAP 认证 → MSK ▼ PMK (Pairwise Master Key) ← 32字节根密钥, 双方都有, 不在空中传 │ │ 4-Way Handshake: PMK + ANonce + SNonce + 双方MAC ▼ PTK (Pairwise Transient Key) ← 本次会话专用, 每次关联重派生 │ ├── KCK (Key Confirmation Key) → 给 EAPOL-Key 帧签 MIC (证明算对了 PTK) ├── KEK (Key Encryption Key) → 加密下发 GTK └── TK (Temporal Key, 临时密钥) → 加密单播数据帧 (CCMP 用这个) GMK (Group Master Key) ← AP 本地生成 │ ▼ GTK (Group Temporal Key) → 加密组播/广播, 整个 BSS 共用, M3 里用 KEK 加密下发

几个关键设计思路，理解了就不用死记：

  • PMK 不在空中传。它是根密钥，PSK 模式下双方从密码各自算出来，802.1X 模式下由 RADIUS 安全下发。空中传的是它的派生输入（nonce），不是 PMK 本身——这是防抓包破解的核心。
  • PTK 每次会话新鲜。同一个 PMK，每次关联用不同的随机 nonce 派生出不同的 PTK。这叫前向保密的弱化版——某次 PTK 泄露不等于 PMK 泄露，其他会话不受影响。
  • PTK 拆三段各司其职。KCK 管认证（签握手帧）、KEK 管密钥分发（加密 GTK）、TK 管数据加密。一把钥匙开一把锁，防止某段被破连累全部。
  • 单播用 PTK、组播用 GTK。为什么组播要单独一把？因为组播帧要让 BSS 内所有 STA 解密，必须用大家共用的钥匙。但单播必须每对独立，一个 STA 的密钥被破不能连累别人。
桥梁 / From BT

BLE 的安全管理（Security Manager）也有一套类似层级：IRK（身份密钥）、LTK（长期密钥）、STK（短期密钥）、会话密钥。思路和 WPA 的 PMK→PTK 是同构的：根密钥长期不变，会话密钥每次配对派生。区别：BLE 的配对（pairing）过程对应 WPA 的 4-Way Handshake，但 BLE 用的是 ECDH + 数值比较/密钥输入，WPA2 用的是"双方已有 PMK，靠 nonce 派生"。你做过 BLE 配对状态机，这套逻辑不陌生。

4-Way Handshake：四个 EAPOL-Key 帧的完整序列

密钥层级讲清了，现在看PTK 到底怎么派生出来。这就是 4-Way Handshake——四个 EAPOL-Key 帧的来回，把"双方各自有 PMK"变成"双方算出同一个 PTK 并互相确认"。逐帧拆：

STA (手机) AP (路由器) │ │ │ 关联已完成, 双方都已有 PMK │ │ │ │ M1 │ │ ◀──────── EAPOL-Key M1 ──────────── │ AP 生成 ANonce │ (ANonce) │ 此时 AP 还不能算 PTK (缺 SNonce) │ │ │ STA 收到 ANonce, 自己生成 SNonce │ │ STA 现在能算 PTK 了! │ │ PTK = PRF(PMK, ANonce, SNonce, 双方MAC) │ │ │ │ M2 │ │ ────────── EAPOL-Key M2 ──────────▶ │ AP 收到 SNonce, 现在也能算 PTK │ (SNonce, MIC) │ AP 验证 MIC = STA 算对了 PTK ✓ │ │ │ M3 │ │ ◀──────── EAPOL-Key M3 ──────────── │ AP install PTK │ (加密的 GTK, RSN-IE, MIC) │ 用 KEK 加密 GTK 下发 │ │ MIC 证明 AP 也算对了 PTK │ STA install PTK + GTK │ │ │ │ M4 │ │ ────────── EAPOL-Key M4 ──────────▶ │ 纯确认, 无新信息 │ (MIC) │ AP 收到 = 一切就绪 │ │ │ ═══ 双方都已授权, 开始加密通信 ═══ │ │ ═══════ 加密数据帧 (Protected=1) ═══▶│

每一步的关键点：

  • M1（AP→STA，发 ANonce）：AP 生成随机数 ANonce 发过去。注意此刻 AP 还算不出 PTK（缺 SNonce），只有 STA 收到后双方凑齐了输入。为什么要 AP 先发？因为 AP 是认证方（authenticator），由它发起握手。
  • M2（STA→AP，发 SNonce + MIC）：STA 生成 SNonce，此时 STA 第一个算出 PTK。M2 里带上 SNonce 和用 KCK 算的 MIC。AP 收到后也能算 PTK 了，验证 MIC = STA 算对了。这一步是"我证明我算出来了"。
  • M3（AP→STA，发加密 GTK + MIC）：AP 确认 STA OK，把组播密钥 GTK 用 KEK 加密下发，再带个 MIC 证明 AP 这边也算对了。STA 收到后 install PTK 和 GTK。
  • M4（STA→AP，纯 MIC 确认）：STA 确认收到并安装了密钥，没有新信息。AP 收到 M4 就知道一切就绪，双方进入"已授权"状态。
为什么要四步，不是三步或两步

核心难点：密钥还没建立时，怎么安全地传密钥协商信息？（鸡生蛋问题）。WPA 的解法：

  • M1/M2 交换 nonce（明文传，但 nonce 不是密钥，泄露无所谓）
  • 用双方都已有但不在空中传的 PMK，加上 nonce 各自算 PTK
  • 用算出的 PTK 里的 KCK 签 MIC，用 MIC 反向证明"我算出了正确 PTK"（不知道 PMK 的人算不出 PTK，签不出 MIC）

所以 M2 的 MIC 是 STA 在说"我知道 PMK"，M3 的 MIC 是 AP 在说"我也知道"，M4 是双向确认。这是用已有根密钥派生会话密钥 + 用 MIC 做身份证明的经典设计。

EAPOL-Key 帧是明文在空中传的（加密还没建立，没法加密）。但攻击者抓到 M1-M4 也拿不到 PTK——因为 PTK 的关键输入 PMK 从来没在空中出现过。这就是为什么抓 4-Way Handshake 只能靠"离线爆破 PMK"（WPA2-PSK 的软肋）。

两种认证方式：PSK vs 802.1X

4-Way Handshake 是固定的，不管哪种 WPA 模式都要走。区别在于PMK 从哪来：

WPA2-Personal (PSK) — 家庭 STA: 密码 ──PBKDF2(4096轮)──▶ PMK AP: 密码 ──PBKDF2(4096轮)──▶ PMK 双方各自从密码算出同一个 PMK, 不需要 RADIUS ↓ 4-Way Handshake (M1-M4) WPA2-Enterprise (802.1X) — 企业 STA ◀──EAP(TLS/PEAP/TTLS)──▶ AP ◀──▶ RADIUS STA 和 RADIUS 服务器做 EAP 认证, AP 当中间人 认证成功, RADIUS 把 MSK 给 AP AP: MSK ──截取──▶ PMK STA: MSK ──截取──▶ PMK ↓ 4-Way Handshake (M1-M4)
模式PMK 来源谁参与优势劣势
WPA2-Personal (PSK) 密码经 PBKDF2 派生 STA + AP 简单，无额外服务器 弱密码可离线爆破；所有用户共享一个密钥
WPA2-Enterprise (802.1X) EAP/RADIUS 认证产生 MSK → PMK STA + AP + RADIUS 每用户独立认证、支持证书、密钥集中管理 需要 RADIUS 基础设施，部署复杂

做固件的你最常碰的是 PSK（家用设备、IoT）。802.1X 主要在企业/校园网，IoT 设备少见，但 802.1X 的 EAP-TLS 在某些工业 IoT 场景（设备证书认证）会用到。

WPA2 vs WPA3：WPA2 软在哪，WPA3 怎么修

WPA2-PSK 有两个著名软肋，WPA3 针对性修了。理解这个对比，你才知道为什么新固件该支持 WPA3：

问题WPA2-PSKWPA3-Personal (SAE)
离线字典攻击 PMK = f(密码)。抓到 4-Way Handshake，回家用字典离线爆破密码（aircrack-ng/hashcat） SAE 派生的 PMK 高熵且每会话新鲜，抓包无法离线验证密码猜测
前向保密 无。密码泄露 → 所有历史抓包可解 有。每次会话密钥独立，密码泄露不连累历史
Deauth 攻击 PMF 可选（通常关）。伪造 Deauth 帧就能踢人下线 PMF 强制。管理帧有加密保护，伪造不了
弱密码 8 位以下密码极易爆破 强制密码 ≥12 位（WFA 认证要求）
SAE (Dragonfly) 的核心改动

WPA3-Personal 用 SAE (Simultaneous Authentication of Equals) 替代 PSK 的密码派生。SAE 基于 Dragonfly 握手（PAKE 协议类），核心思路：

  • 双方用密码在椭圆曲线上派生一个 Password Element (PWE)
  • 做 COMMIT/CONFIRM 交换，派生出 PMK
  • 关键：PMK 不再直接等于 f(密码)，而是加入了随机椭圆曲线运算，高熵且每会话新鲜

结果：攻击者抓到 SAE 交换，无法离线验证密码猜测（每次尝试都要和 AP 在线交互，会被限速）。这从根本上修掉了 WPA2-PSK 的离线爆破软肋。

SAE 也有代价：椭圆曲线运算重（CPU 开销），而且 Dragonblood 研究发现了侧信道类攻击（PWE 派生的迭代次数泄漏信息）。但整体上 WPA3 显著强于 WPA2，是新固件的安全方向。

加密协议：WEP → TKIP → CCMP

4-Way Handshake 派生出密钥后，用什么算法加密数据帧？这是另一条演进线：

协议算法状态致命问题
WEP (1997) RC4 + 静态密钥 2004 废弃 IV 24bit 会复用，几分钟可破
TKIP (WPA1, 2003) RC4 + 每包密钥混合 + Michael MIC 2012 废弃 过渡补丁，Michael MIC 可被碰撞攻击
CCMP (WPA2, 2004) AES（CTR 加密 + CBC-MAC 完整性） 现行标准 —（AEAD，每帧一次性 nonce）

CCMP 是 WPA2/3 的核心：用 AES 同时做加密（CTR 模式）和完整性校验（CBC-MAC），合称 AEAD（Authenticated Encryption with Associated Data）。每帧用一次性 nonce，杜绝了 WEP 的密钥复用漏洞。Wi-Fi 4/5/6 的数据帧加密都用 CCMP。做新固件只支持 CCMP，TKIP/WEP 当历史了解就行。

WEP 是"加密协议设计反面教材"，也是第 7 课那个"802.11 Authentication ≠ 安全认证"名字陷阱的历史源头：WEP 时代 802.11 Auth 帧原本要做 Shared Key 认证（带挑战-应答），WEP 被废后这步就空壳了，但名字和帧格式留了下来。

PMF：管理帧也要保护

CCMP 保护的是数据帧。但管理帧（Deauth、Disassociate、部分 Action 帧）在 WPA2 默认是不加密的——这有个大问题：攻击者伪造一个 Deauth 帧就能把 STA 踢下线，这是早年 Wi-Fi 最常见的攻击（咖啡店里"WiFi 突然断开"很多是这个）。

PMF (Protected Management Frames, 802.11w) 给管理帧加加密保护和完整性校验：WPA2 可选（多数关着），WPA3 强制开启。做新固件上 WPA3，PMF 必须支持。

对驱动栈：安全机制落在代码哪里

对驱动重要 / For Driver Stack

安全在 Wi-Fi 协议栈里的代码分布：

  • 职责边界：wpa_supplicant（用户态）做 4-Way Handshake、密钥派生、PMK/PTK 管理；mac80211 + 驱动负责把密钥 install 到硬件（硬件加密加速）。两者通过 nl80211 的 NL80211_CMD_NEW_KEY 通信。
  • 硬件加密 offload：绝大多数 Wi-Fi 芯片的加密/解密在硬件里做（CCMP 计算量大，软件做费 CPU）。驱动要把 PTK/GTK 写进芯片的 key table，芯片在 tx/rx 时自动加解密。这是性能关键路径。
  • Protected Frame bit：第 6 课讲的 Frame Control 里这一位，置 1 表示帧体已加密。驱动 rx 路径看这一位决定要不要解密。填错会导致"收到加密帧当明文解析 → 全是乱码"。
  • 4-Way Handshake bug：最常见的是握手超时——Reason Code 15。原因：nonce 生成错误、PMK 派生错误（密码/SSID 拼接错）、MIC 计算错误、密钥 install 时机错。调试看 wpa_supplicant -ddd 日志的 M1-M4 来回。
  • 抓包验证：抓包能看到 EAPOL-Key M1-M4，但看不到 PMK/PTK 本身。验证密钥正确性要靠 Wireshark 配置 PMK（仅 PSK 且已知密码时）来解密数据帧。
  • SAE/TWT 等新特性：WPA3 的 SAE、802.11w 的 PMF，新固件要确认芯片固件版本支持。老的只支持 WPA2 的芯片不能直接上 WPA3。

自检：安全机制吃透了吗

这课的钉子：密钥层级 + 4-Way Handshake

学完这课，你脑子里应该有 Wi-Fi 安全的两张图：

  1. 密钥金字塔：密码 → PMK（根）→ PTK（会话，拆 KCK/KEK/TK）+ GTK（组播）
  2. 4-Way Handshake：M1 发 ANonce → M2 回 SNonce+MIC → M3 发 GTK+MIC → M4 确认

还有一条演进线：WEP（废弃）→ TKIP（废弃）→ CCMP（现行）；PSK（可爆破）→ SAE（抗爆破）；PMF（可选）→ PMF（WPA3 强制）。

你现在能做的：

  • 看到抓包里的 EAPOL-Key M1-M4，能还原出握手到哪一步
  • 理解为什么关联成功还不能传数据（要 4-Way Handshake 完成进"已授权"）
  • 区分 PSK / 802.1X / SAE 三种 PMK 来源
  • 知道 WPA2 软在哪、WPA3 怎么修
  • 定位 4-Way Handshake 超时（Reason 15）该看哪里
