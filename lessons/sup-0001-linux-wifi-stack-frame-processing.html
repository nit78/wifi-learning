Linux Wi-Fi 栈 —— 三类帧各在哪一层处理

这是回答你临时提问的课外补充，不打断协议主线。主线第 8 课仍按计划讲协议内容（省电模式）。

这个问题命中你转 Wi-Fi 驱动/协议栈方向最该先建的一张全景图。前面七课讲的都是协议本身（帧长什么样、状态机怎么走、CSMA/CA 怎么退避），这篇讲的是这些协议机制在 Linux 上分别落在哪段代码里 —— 数据帧 (Data frame)、管理帧 (Management frame)、控制帧 (Control frame) 各自由谁收发、谁解析、谁对用户态可见。

这张图一旦立起来，你后面看任何 Wi-Fi 驱动（ESP32、Realtek、MTK、高通）都能立刻定位："这个 bug 在哪一段"。这是 BT 驱动栈经验最直接的迁移点 —— 你做 BT 时知道 HCI/ACL/SCO 各在哪一层，Wi-Fi 这边只是换了名字和分层方式。

支点：一个数据帧穿越四层软件栈

本课支点 / Core Insight

Linux Wi-Fi 软件栈分四层，从下到上：

  1. 固件 (Firmware) —— 跑在 Wi-Fi 芯片（"模组"）内部的小操作系统上，碰硬件寄存器、RF、DMA。对内核是个黑盒，通过驱动定义的命令接口（PCIe/SDIO/USB 上的厂商私有协议）通信。
  2. 设备驱动 (Device driver) —— 内核态（drivers/net/wireless/），把固件接口"翻译"成 Linux 网络栈认得的形式。ath10k / iwlwifi / brcmfmac 等都是这一层。
  3. mac80211 + cfg80211 —— 内核态（net/mac80211/ + net/wireless/），802.11 协议框架。mac80211 实现 MLME (MAC 子层管理实体)、帧封装、加密、聚合；cfg80211 是统一配置入口。
  4. 用户态 (Userspace) —— wpa_supplicant（STA 侧）、hostapd（AP 侧），通过 nl80211 (netlink) 与内核通信。

三类帧的处理职责，就分布在这四层里。数据帧几乎一路上下贯通；管理帧在中间被处理；控制帧绝大多数在固件里就消化了，内核根本看不到。

先看一张全景图，把四层竖起来：

桥梁 / From BT

BT 的分层你应该熟：App → Host Stack (L2CAP/ATT/SM) → HCI → Controller 固件 → RF。Wi-Fi 这边一一对应：wpa_supplicant ≈ 上层协议栈；nl80211 ≈ HCI（用户态↔内核的控制通道）；mac80211 + 驱动 ≈ Host Controller Driver；芯片固件 ≈ Controller 固件。最大的区别：BT 的 MLME 永远在 controller，Wi-Fi 的 MLME 可以在内核（SoftMAC）也可以在固件（FullMAC） —— 这就是下一节的分水岭。

分水岭：SoftMAC vs FullMAC —— MLME 跑在哪

在回答"三类帧各在哪处理"之前，必须先知道一个根本的架构分叉：MLME (MLME, MAC 子层管理实体) —— 也就是管理帧的收发状态机 —— 到底跑在哪里。这决定了后面所有的"谁处理什么"。

SoftMAC vs FullMAC 的定义

SoftMAC (Software MAC, 软件 MAC)：MLME 放在内核软件里做，由 mac80211 框架统一实现。驱动只管把芯片收到的 802.11 帧（含管理帧）原样上送，mac80211 在软件里跑状态机、解析管理帧、构造 Auth/Assoc Request。

FullMAC (Full MAC, 全 MAC)：MLME 放在芯片固件里做。内核看不到裸管理帧 —— 它只通过 cfg80211 收发"高级命令"（scan / connect / key），固件内部完成整个帧交换，再把结果（"连上了"/"扫描到 3 个 BSS"）上报。

MLME 跑在哪 内核 mac80211（net/mac80211/mlme.c） 芯片固件里
驱动注册方式 实现 ieee80211_ops → ieee80211_register_hw() → mac80211 替它向 cfg80211 注册 实现自己的 cfg80211_ops → wiphy_new_nm() → cfg80211_register_wiphy()，绕过 mac80211

常见形态 PCIe / MiniPCIe（笔记本、AP、路由器主芯片） SDIO / USB 模块（IoT、树莓派、外置网卡）

⚠️ 最常踩的坑：ath10k / ath11k / iwlwifi 不是 FullMAC

很多人因为这些驱动固件很"重"（做扫描 offload、连接管理、加密 offload、聚合），就误以为它们是 FullMAC。架构上它们都是 SoftMAC —— MLME 仍然走 mac80211。判断依据不是"固件干了多少活"，而是"MLME 跑在哪、驱动注册走不走 mac80211"。ath11k 官方文档原文就写"ath11k uses mac80211"。

理解了这个分水岭，下面这张表就是本课的核心交付物。

下面把这三行逐一展开。

① 数据帧：贯通四层的流水线

数据帧是唯一一路贯通所有层的帧类型。发送方向（应用 → 空中）的完整路径：

应用 socket │ write/send ▼ 内核网络栈 (TCP/IP → netfilter → 路由) │ 到 wlan0 接口的 sk_buff ▼ mac80211 ieee80211_tx → __ieee80211_tx │ ① 802.3 以太网头 → 802.11 MAC 头（加 ToDS/FromDS、地址映射） │ ② 速率控制算法选 MCS（Minstrel HT 或固件 offload） │ ③ 软件加密（若没硬件 offload）/ 分片 ▼ 设备驱动 ath10k_tx / iwl_tx │ 把 sk_buff 塞进驱动队列，翻译成厂商固件命令 ▼ 固件 / 硬件 │ ④ 硬件加密（CCMP/GCMP） ⑤ A-MPDU 聚合 ⑥ 功率/校准 │ ⑦ RF 发射，按 CSMA/CA + IFS + backoff 抢介质 ▼ 📡 空中 (Air)

接收方向（空中 → 应用）是这条线的镜像，但有几个关键点要记住：

对驱动重要 / For Driver Stack

数据帧 RX 路径的关键函数链（都在 net/mac80211/rx.c）：

  • ieee80211_rx() → ieee80211_rx_list()：驱动收到 MPDU 后的入口（注释原文："called by a low level driver when an 802.11 MPDU is received from the hardware"）。
  • ieee80211_rx_handlers()：一条由 CALL_RXH 宏串起来的处理链，依次执行一系列 ieee80211_rx_h_* 处理器。
  • 典型顺序：ieee80211_rx_h_check（查 STA）→ ieee80211_rx_h_decrypt（软/硬解密）→ ieee80211_rx_h_defragment（分片重组）→ ieee80211_rx_h_sta_process → ieee80211_rx_h_ampdu（A-MPDU 重排）→ ieee80211_rx_h_data（剥 MAC 头转 802.3）。
  • 最后 netif_receive_skb() 把以太网帧送上内核网络栈，和有线网卡合流。

这就是为什么 数据帧不经 wpa_supplicant —— 数据路径和协议控制路径在内核就分叉了，数据帧走网络栈直接到 socket，wpa_supplicant 只管控制平面（密钥/连接）。

⚠️ 版本提示：ieee80211_rx_h_passive_fill 这类处理器在新内核里已被合并/移除。要看你手上内核版本的确切处理链，打开 net/mac80211/rx.c 搜 CALL_RXH 就是当前真相。处理链的架构（驱动→mac80211→netif）跨版本稳定，具体函数名会演进。

② 管理帧：内核 mac80211 的主战场

管理帧是你做协议栈移植最直接对口的部分 —— 这正是 BT 协议栈状态机经验镜像迁移的地方。SoftMAC 驱动下，STA 模式的管理帧 MLME 在内核 mac80211 里跑（mac80211 官方文档原文："In mac80211 the MLME is done in the kernel for station mode (STA) and in userspace for AP mode (hostapd)."）。

STA 模式管理帧 固件 mac80211 (mlme.c) wpa_supplicant
Beacon / Probe
(扫描)		 RF 收 + DMA（可能代发 Probe Request 做扫描 offload） ieee80211_rx_h_mgmt 收到后入队，mlme.c 解析 BSS 信息上报 cfg80211 发 NL80211_CMD_TRIGGER_SCAN，收 BSS 列表结果
Authentication
(第 7 课 State 1→2)		 RF 收发 + DMA mac80211 构造/解析 Auth Seq 1/2，推进状态机（不是 wpa_supplicant 做） 发"connect"命令，状态机由内核跑，wpa_supplicant 等结果
Association
(State 2→3)		 RF 收发 + DMA mac80211 构造/解析 Assoc Req/Resp，读 AID，进 State 3 同上，等关联结果
EAPOL-Key
(4-Way Handshake)		 RF 收发 + DMA 当作数据帧上送（EAPOL 走数据帧通道） wpa_supplicant 在这里接管：派生 PTK/GTK，装密钥到内核
Deauth / Disassoc RF 收发 + DMA mac80211 处理，把状态机打回 State 1/2，上报事件 收到断连事件，决定重连或换 AP（漫游）
关键边界：mac80211 管 State 1→2→3，wpa_supplicant 管密钥

这是你移植时必须尊重的职责边界：

  • 认证 + 关联（State 1→2→3）：在 SoftMAC 驱动上是内核 mac80211 跑的（net/mac80211/mlme.c 里的状态机）。
  • 4-Way Handshake / 密钥派生：在用户态 wpa_supplicant 里跑（PSK/802.1X 在这里）。
  • AP 模式 MLME：在用户态 hostapd 里跑（mac80211 不管 AP 的状态机）。

这个分界让 MAC 层和安全层解耦 —— 同一套 mac80211 状态机能挂 Open / WPA2-PSK / WPA2-Enterprise / WPA3。调试关联问题时，同时看内核日志（dmesg）和 wpa_supplicant 日志（-ddd），按时间戳对齐，问题在哪一段一眼看出。

FullMAC 驱动下这张表会大幅简化：固件自己跑完整个 MLME，内核只通过 cfg80211 收发高级命令（scan / connect），mac80211 整层不参与。这也是为什么 FullMAC 驱动代码量通常比 SoftMAC 小（状态机都在固件里），但调试更黑盒（管理帧细节对内核不可见，得靠固件的事件日志）。

③ 控制帧：固件独占，内核几乎看不见

这是新人最意外的一行：ACK、CTS、BlockAck 这些控制帧，内核基本完全看不到。原因是个硬物理约束：

为什么控制帧必须在固件/硬件做

ACK、CTS、BlockAck 都是控制响应帧，必须在收到原帧后的 SIFS（短帧间间隔，~16 μs）内发出。这个时间预算里，光是把帧从 PCIe/SDIO 传到内核、走中断处理、再让 CPU 决定回个 ACK 然后传回芯片发射 —— 根本来不及。SIFS 的物理含义（第 5 课）就是"短到只有 RF 收发链路自己来得及响应"。所以这些帧天生属于芯片 MAC 硅片 + 固件。

所以 mac80211 对控制帧的态度是"我只看后果，不看帧本身"：

控制帧 固件 / 硬件 mac80211
ACK 硬件在 SIFS 内自动回 ACK；TX 侧把"有没有收到 ACK"记进 TX 状态 读 TX 状态里的 ACK 位 (IEEE80211_TX_STAT_ACK)，没收到 → 触发重传/速率降阶
RTS / CTS 隐藏节点握手全在固件做（mac80211 只设个"启用 RTS 阈值"的参数） 不参与，只在配置里设 RTS/CTS 阈值
BlockAck
(A-MPDU 用)		 ADDBA/DelBA 帧、BlockAck 帧的收发都在固件 会话簿记：ieee80211_start_tx_ba_session() / ieee80211_start_rx_ba_session() 建立重排缓冲区，但帧本身不上送内核

mac80211 RX 处理链里有一句注释印证了这点：到管理帧处理器时，"数据帧和控制帧已经被处理掉了"（"Data and control frames are already handled"）—— 意思是控制帧在到达 mac80211 的帧解析逻辑之前，就被固件消化了。

桥梁 / From BT

BT 里也是一样的逻辑：BLE 的 CRC 校验、接入地址匹配、空包 (empty PDU) 自动应答都在 controller 硅片做，host 根本不参与。Wi-Fi 的 ACK/CTS 之于 mac80211，就像 BLE 的链路层自动应答之于 HCI host —— 都是时序逼出来的硬件/固件职责。你做 BT 时从来没在 host 上"手动回 ACK"，Wi-Fi 这边同理。

FullMAC 视角：管理帧也下沉到固件

上面三节讲的是 SoftMAC（mac80211）路径。如果你将来做的是 FullMAC 驱动（比如某个 SDIO/USB Wi-Fi 模块移植），管理帧的处理位置会变：

FullMAC 路径 固件 cfg80211（无 mac80211） 用户态
管理帧 固件自己跑完整 MLME（Beacon/Auth/Assoc 都在固件里交换） 只通过 cfg80211_ops（驱动自己实现的）收发高级命令：scan / connect / add_key wpa_supplicant 同样经 nl80211，但发的是"connect"命令而不是"authenticate + associate"
数据帧 固件做更多封装/加密/聚合（因为 mac80211 不在） 驱动直接把帧上送网络栈（netif_receive_skb），不经 mac80211 同 SoftMAC，走 socket
控制帧 完全在固件（同 SoftMAC） 不参与 不可见

判断一个驱动是 SoftMAC 还是 FullMAC，最直接的方法：看它是否在 drivers/net/wireless/ 下调用 ieee80211_register_hw()（SoftMAC）还是 cfg80211_register_wiphy()（FullMAC）。前者依赖 mac80211，后者绕开它。

对驱动栈：这套架构落在代码哪里

对驱动重要 / For Driver Stack

把本课的分层映射到内核源码路径（可在 Elixir Bootlin 在线读）：

  • net/mac80211/ —— SoftMAC 框架核心
    • rx.c —— 数据/管理帧 RX 路径 + ieee80211_rx_handlers 处理链
    • tx.c —— 数据帧 TX 路径（ieee80211_tx）
    • mlme.c —— STA 模式 MLME 状态机（认证/关联/扫描的核心，你的协议栈经验落点）
    • cfg.c —— mac80211 自己向 cfg80211 注册的那份 cfg80211_ops
  • net/wireless/ —— cfg80211 + nl80211
    • core.c —— cfg80211 核心、wiphy 注册
    • nl80211.c —— netlink 命令处理器（NL80211_CMD_*）
  • include/net/cfg80211.h —— struct cfg80211_ops、wiphy 定义（FullMAC 驱动的契约）
  • include/net/mac80211.h —— struct ieee80211_ops、ieee80211_rx() 等 SoftMAC 驱动的契约
  • drivers/net/wireless/... —— 各厂商驱动（ath10k/ath11k/iwlwifi/brcmfmac/mwifiex...）

调试黄金组合（关联/吞吐/掉线问题通用）：

  • wpa_supplicant -ddd —— 看用户态状态机和密钥流程
  • dmesg / iw event —— 看内核 cfg80211/mac80211/驱动的事件
  • monitor mode 抓包（tcpdump / Wireshark）—— 看真实空中帧（这是 FullMAC 黑盒调试的唯一外部视角）
  • iw dev wlan0 station dump —— 看实时链路状态（MCS、信号、重传率）

三路日志对时间戳，问题落在固件/驱动/mac80211/用户态哪一段，基本一眼定位。

自检：分层吃透了吗

这课的钉子：四层分工图是 Wi-Fi 驱动的地图

学完这课，你脑子里应该有一张贯穿四层的分工图，并且能对任何一类帧说出"它落在哪一段"：

  1. 数据帧：贯通四层 —— 应用→网络栈→mac80211（封装/加密）→驱动→固件（聚合/功率/RF）。RX 路径在 mac80211 走 ieee80211_rx_handlers 处理链，最后 netif_receive_skb 上送网络栈。不经 wpa_supplicant。
  2. 管理帧：SoftMAC 下 STA 模式 MLME 在内核 mac80211（mlme.c）；AP 模式在用户态 hostapd；FullMAC 下在固件。wpa_supplicant 只管密钥（4-Way Handshake）和漫游决策。
  3. 控制帧：几乎全在固件/硬件（SIFS 时序逼出来的），内核只看 ACK 成功位、维护 BlockAck 会话簿记。

这张图把前七课的协议知识锚定到了代码：

  • 第 5 课的 CSMA/CA/IFS/backoff → 固件 MAC 硅片（SIFS 逼出来的）
  • 第 6 课的 MAC 帧结构 → mac80211 的 rx.c/tx.c 封装解析
  • 第 7 课的 State 1/2/3 状态机 → mac80211 的 mlme.c（SoftMAC）或固件（FullMAC）
  • 4-Way Handshake → wpa_supplicant 用户态
桥梁 / From BT —— 分层完全同构

把这张图和 BT 对照，你会发现分层思想完全一致，只是名字换了：

  • Wi-Fi nl80211 ≈ BT HCI（控制平面通道）
  • Wi-Fi mac80211 ≈ BT Host Controller Driver
  • Wi-Fi wpa_supplicant ≈ BT 上层协议栈（SM 做密钥，对应 wpa_supplicant 做 4-Way）
  • Wi-Fi 芯片固件 ≈ BT Controller 固件

最大的架构差异：BT 的 MLME 永远在 controller（LL 状态机跑在硅片），而 Wi-Fi 的 MLME 可以在内核（SoftMAC）也可以在固件（FullMAC） —— 这是 Wi-Fi 比 BT 多出来的一个设计自由度，也是你选驱动架构时要做的第一个决定。
