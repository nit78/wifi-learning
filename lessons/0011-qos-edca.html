QoS / EDCA —— 四个优先级怎么排出来

第 5 课讲了 CSMA/CA 的基础版 DCF——所有 STA 平等竞争，CWmin=15/CWmax=1023，没有优先级。问题是：Voice 帧和 Background 下载帧排一队竞争，语音可能被挤得卡顿。802.11e（2005）就是来修这个的：把数据分成 4 个优先级，让语音/视频优先抢信道。

这课对你做驱动的落点很直接——tx 队列按 TID 分流就是 EDCA 的实现。你做 BT 协议栈时按 ACL 链路类型分流的经验直接迁移，只是 Wi-Fi 多了 AIFS/CW/TXOP 三套旋钮。

支点：EDCA 用三道旋钮排出优先级

本课支点 / Core Insight

EDCA (Enhanced Distributed Channel Access) 是 802.11e 的 QoS 版 DCF。它没有引入中央调度（还是分布式竞争），而是给每个 AC（接入类别）配三套参数，让高优先级 AC 抢到信道的概率更大：

  1. AIFS（第一道闸门）：发之前要等的时间。语音等最短，背景等最长。
  2. CWmin/CWmax（第二道闸门）：退避窗口范围。语音最小，背景最大。
  3. TXOP（第三道闸门）：抢到信道后能连续发多久。语音/视频有配额，普通数据只发一帧。

三道闸门协同：高优先级 AC 等得短、退避少、发了能多发。低优先级相反。这是概率上的"优先"，不是绝对优先——语音仍可能被背景帧碰撞，只是概率小得多。

四个 AC：优先级从高到低

802.11e 定义 4 个 AC (Access Category)，对应 4 种业务类型：

AC全名典型业务特点
AC_VO Voice VoIP、视频通话语音 小包、低延迟、双向周期，对延迟/抖动最敏感
AC_VI Video 流媒体视频、IPTV 中大包、连续流，延迟敏感但容忍少量丢包
AC_BE Best Effort 网页、邮件、聊天 普通上网，尽力而为，不保证延迟
AC_BK Background 下载、备份、同步 后台流量，最低优先级，延迟无所谓
桥梁 / From BT

BLE 没有完全对应的优先级机制（BLE 是 master 调度的，slave 按分配的 slot 发，不存在竞争）。但 BT 经典的 ACL 链路有类似思路：DH/DM 链路类型、flush timeout、QoS setup 请求，都是在区分业务优先级。Wi-Fi 的 AC 是更彻底的版本——把优先级刻进介质访问本身（AIFS/CW），而不是靠上层调度。你做 sub-GHz 协议栈时如果做过"应急报文优先于周期上报"，那是上层优先级，和 EDCA 的 MAC 层优先级不冲突，可以叠加。

第一道闸门：AIFS

第 5 课讲 DCF 里所有 STA 都等 DIFS = SIFS + 2×SlotTime。EDCA 把这个固定值换成按 AC 变化的 AIFS：

AIFS[AC] = SIFS + AIFSN[AC] × SlotTime ↑ AIFSN ≥ 2 (AC 自己决定要等几个 slot) 802.11a/g (SlotTime=9μs, SIFS=16μs) 默认值: AC_VO: AIFSN=2 → AIFS = 16 + 2×9 = 34 μs AC_VI: AIFSN=2 → AIFS = 16 + 2×9 = 34 μs AC_BE: AIFSN=3 → AIFS = 16 + 3×9 = 43 μs AC_BK: AIFSN=7 → AIFS = 16 + 7×9 = 79 μs 对比 DCF 的 DIFS = 16 + 2×9 = 34 μs → AC_VO/VI 的 AIFS = DIFS，AC_BE/BK 比 DIFS 更长

关键：AIFS 是必等时间，是 EDCA 优先级的第一道闸门，比 CW 更重要。原因——退避计数器在 AIFS 结束后才开始递减，AIFS 长的 AC 根本来不及开始退避，信道可能已经被 AIFS 短的 AC 抢走。所以 AC_VO（等 34μs）几乎总是在 AC_BK（等 79μs）之前有机会开始竞争。

第二道闸门：CWmin/CWmax

AIFS 决定"什么时候开始竞争"，CW 决定"竞争时退避多久"。第 5 课讲 DCF 的 CW 是 [15, 1023]。EDCA 给不同 AC 不同的 CW 范围：

ACCWminCWmax对比 DCF (15/1023)
AC_VO37窗口最小，退避最短，最容易抢到
AC_VI715较小
AC_BE151023和 DCF 一样
AC_BK151023和 DCF 一样（但 AIFS 更长，所以仍更低优先）

注意 AC_BE 和 AC_BK 的 CWmin/CWmax 数值相同（都是 15/1023）——它们的优先级差异完全靠 AIFS 区分（AC_BK 的 AIFSN=7 比 AC_BE 的 3 长很多）。这正好说明 AIFS 是更基础的闸门。

内部碰撞 (Internal Collision)

EDCA 特有的现象：同一个 STA 内部，多个 AC 同时退避计数到 0，因为它们独立计数。怎么决定谁先发？

规则：高优先级 AC（AIFS 短/CW 小）胜出发送，低优先级 AC 当成发生碰撞一样 CW 倍增。这是 EDCA 在单设备内部实现优先级的方式——4 个 AC 像同一个 STA 里的 4 个虚拟竞争者，内部先比一轮。

第三道闸门：TXOP

前两道闸门管"谁先抢到信道"，TXOP 管"抢到之后能发多久"。DCF 里抢到信道只发一个帧（DATA→ACK 就结束），下次还要重新竞争。EDCA 给高优先级 AC 一个 TXOP (Transmission Opportunity, 传输机会) 配额，在这个窗口内可以连续发多个帧不必重新竞争：

ACTXOP (802.11a/g)含义
AC_VO1.504 ms连续发约多个语音帧（语音每帧 20ms 周期，一次发不完）
AC_VI3.008 ms连续发视频数据块
AC_BE0只能发一帧（和 DCF 一样）
AC_BK0只能发一帧

TXOP=0 意味着 Contention-Free Bursting 关闭，每发一帧都要重新竞争。TXOP>0 时可以在窗口内连发（burst），对语音/视频这种连续小包尤其重要——省掉了每个帧之间的重复竞争开销。

TXOP 值的单位是 32μs 的整数倍（历史原因）。AC_VO 的 1.504ms = 47 × 32μs，AC_VI 的 3.008ms = 94 × 32μs。抓包看 EDCA Parameter Set 元素时会看到这些倍数。

四道闸门协同：一张图看清

时间 ─▶ │ ←─ SIFS (16μs) ─→│←─ SlotTime ─→│ │ │ 1 2 3 4 5 6 7 AC_VO 发! 发! 发! 发! 发! ← AIFSN=2, 2 slot 后就能竞争 AC_VI 发! 发! 发! 发! 发! ← AIFSN=2, 同 AC_VO AC_BE . 发 发 发 ← AIFSN=3, 3 slot 才开始（晚一个 slot） AC_BK . . . . . . . . 发 ← AIFSN=7, 7 slot 才开始（晚很多） │ │ 信道一旦在某 slot 被抢，后面的 AC 都退避重来 │ → AC_VO/VI 几乎总能在 AC_BE/BK 之前抢到

读图：所有 AC 都在等 AIFS，AIFS 结束后开始退避计数（每个 SlotTime 减 1）。AIFSN 越小越早开始计数，越早有机会计数到 0 抢到信道。AC_VO/VI 在第 2 个 slot 就能抢，AC_BK 要等到第 7 个 slot——大多数情况下信道早就被高优先级抢走了。这就是 EDCA"概率优先"的本质。

这张表不用背，但要建立直觉：从 VO 到 BK，AIFS 递增、CW 递增、TXOP 递减——三道闸门同向收紧，优先级就排出来了。

参数谁决定

这些默认值是标准建议，实际值由 AP 在 Beacon 的 EDCA Parameter Set 元素里广播。AP 可以调整（比如企业网给 VoIP 专门的优化参数）。STA 关联后从 Beacon 读这些参数应用到自己的 4 个 AC 队列。你做固件时，解析 Beacon 的 EDCA Parameter Set 并更新本地 AC 参数是必备功能。

TID：上层到 AC 的桥梁

上层（IP 层）怎么告诉 MAC 层"这个帧是语音"？靠 TID (Traffic Identifier, 业务标识符)。链路：

IP 层 DSCP (6 bit, 区分服务码点) ↓ (操作系统/驱动映射) 802.11 QoS Data 帧的 QoS Control 字段: TID (4 bit, 0-7) ↓ (固定映射) 4 个 AC 映射关系 (典型, 标准 HCF 映射): TID 6, 7 → AC_VO (语音) TID 4, 5 → AC_VI (视频) TID 0, 3 → AC_BE (尽力, 每两个 TID 一个 AC) TID 1, 2 → AC_BK (背景)

第 6 课讲过 QoS Data 帧比普通 Data 帧多 2 字节 QoS Control 字段，里面就装 TID。这就是 EDCA 在帧结构上的入口——普通 Data 帧走 DCF（不分 AC），QoS Data 帧走 EDCA（按 TID 分流）。你做驱动 tx 路径时，从 socket 拿到包，按 DSCP/TID 分到 4 个硬件队列，每个队列按对应 AC 的参数竞争——这就是 EDCA 的完整实现。

HCF、EDCA、HCCA、WMM 的关系

这几个名词容易混，理一下：

名词是什么用得多吗
HCF 802.11e 的 QoS MAC 总框架（Hybrid Coordination Function） 是 EDCA + HCCA 的总称
EDCA HCF 的分布式子集（竞争式） 几乎都在用
HCCA HCF 的集中式子集（AP 轮询调度，类似 TDMA） 极少用（AP/STA 实现复杂，没普及）
WMM Wi-Fi 联盟对 EDCA 子集的认证品牌 所有现代设备都过 WMM 认证

记住一条就够：日常说的 "802.11e QoS" = EDCA = WMM。HCCA 当背景知识了解，做固件几乎不会碰到（没设备真正用）。

对驱动栈：EDCA 落在代码哪里

对驱动重要 / For Driver Stack

EDCA 在驱动栈里的落点：

  • 4 个硬件队列：绝大多数 Wi-Fi 芯片有 4 个 tx 驱动队列，对应 AC_VO/VI/BE/BK。驱动把 QoS Data 帧按 TID 映射到对应队列，每个队列的硬件寄存器配 AIFSN/CWmin/CWmax/TXOP。这是 EDCA 的硬件实现路径。
  • Beacon 解析：rx 路径要解析 Beacon 的 EDCA Parameter Set 元素，更新 4 个 AC 的参数。AP 改参数时 STA 要跟着改。SoftMAC 在 mac80211 的 ieee80211_rx_mgmt_beacon 里做；FullMAC 在固件里做。
  • TID 映射：tx 路径从上层（socket/网络栈）拿到帧，按 DSCP → TID → AC 映射分队列。映射规则在 cfg80211 配置，驱动可覆盖。
  • QoS Data 帧构造：发 QoS Data 帧时，Frame Control 的 Type/Subtype 要标成 QoS Data，并在头里加 2 字节 QoS Control 字段（装 TID）。第 6 课的帧结构细节在这里落地。
  • 调试：tx 卡顿/延迟高时，用 iw dev wlan0 station dump 看 tx retries、用 ethtool -S 看按 AC 的统计。语音卡顿常见原因：①TID 映射错（语音进了 AC_BE）；②EDCA 参数没正确从 Beacon 更新；③硬件队列优先级没配；④TXOP=0 导致每次发一帧。

自检：EDCA 吃透了吗

这课的钉子：三道闸门排出概率优先

学完这课，你脑子里应该有 EDCA 的三道闸门骨架：

  1. AIFS（第一道，最基础）：AIFSN=2/2/3/7，等的时间从短到长
  2. CWmin/CWmax（第二道）：3-7 / 7-15 / 15-1023 / 15-1023
  3. TXOP（第三道）：1.504ms / 3.008ms / 0 / 0

还有一条映射链：IP DSCP → TID → AC → 硬件队列，这是 EDCA 从上层到驱动的完整路径。

你现在能做的：

  • 看到 Beacon 的 EDCA Parameter Set，知道 4 个 AC 各自参数
  • 解释为什么 AIFS 比 CW 更基础（必等时间 vs 退避范围）
  • 理解 TID 是上层到 MAC AC 的桥梁
  • 定位语音卡顿：TID 映射错 / EDCA 参数没更新 / 硬件队列没配
  • 区分 EDCA（分布式，用）和 HCCA（集中式，不用）
