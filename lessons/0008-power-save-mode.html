Power Save 省电模式 —— IoT 续航的协议层设计

这一课兑现第 7 课埋的三个伏笔：AID、TIM、PS-Poll。它们不是孤立的字段，而是一整套省电协议机制的核心。这是你做 Wi-Fi IoT 固件（智能门锁、传感器、电池供电设备）绕不开的主题，也和你做 sub-GHz 低功耗协议栈的经验直接对话。

Wi-Fi 省电的根本矛盾：STA 不知道下行什么时候来，只能一直醒着听——但射频一直开，电池几小时就没了。802.11 的解法是把"被动接收"改成"主动预约取"：STA 睡眠时 AP 帮它缓存帧，STA 定期醒来问"有我的快递吗"，有就取，没有继续睡。这一思路从 1997 年的 legacy PS-Poll 一路演进到 Wi-Fi 6 的 TWT，越走越精细。

支点：省电的本质是"睡眠 + 定期取件"

本课支点 / Core Insight

所有 802.11 省电机制都遵循同一个三步循环，只是"取件"的触发方式和睡眠粒度在进化：

  1. 声明睡眠：STA 在某个帧里置 Power Management bit=1，告诉 AP"我要睡了"
  2. 睡眠：关闭射频（这才是真正省电的地方），AP 把发给它的下行帧缓存起来
  3. 醒来取件：STA 醒来，检查 TIM/协商窗口，发 PS-Poll / 触发帧取回缓存的帧

从 legacy PS-Poll → U-APSD → TWT，变的只是第 3 步怎么触发、第 2 步能睡多久。把这三步当成骨架，所有变体都好理解。

桥梁 / From BT & sub-GHz

这正是 BLE 连接事件 (connection event) 的思路：slave 在 anchor point 醒来收发，其余时间睡眠，master 缓存数据。也对应 LoRaWAN 的 Class A——上行后短开两个接收窗口，其余深睡。无线低功耗的核心范式是相通的：协议层预约时间窗口，射频按需开关。你做 sub-GHz 协议栈时设计的"睡-醒-收-睡"状态机，在 Wi-Fi 里换了一套字段名。

① Legacy Power Save：PS-Poll + TIM + DTIM

最古老（802.11-1997）也最基础的省电模式。整套机制围绕 Beacon 周期运转：

三个关键概念，对应第 7 课埋的伏笔：

概念是什么在哪
TIM Beacon 里的位图，每一位对应一个 AID。AID 位=1 表示"AP 为这个 STA 缓存了单播帧" Beacon 的 Information Element
DTIM 特殊的 TIM（按 DTIM Period 周期出现）。不仅指示单播缓存，还立即发送缓存的广播/组播帧。省电 STA 必须在 DTIM 醒来 Beacon（DTIM Count=0 的那个）
PS-Poll 省电 STA 向 AP"取一个缓存帧"的控制帧。Duration/ID 字段放 AID（不是 NAV） 控制帧

DTIM 为什么强制醒来：广播/组播帧（如 ARP、DHCP、mDNS）没有特定的接收方，没法"按 AID 缓存等取"。AP 把它们攒着，等 DTIM Beacon 发完立即广播出去。省电 STA 必须在 DTIM 醒来收，否则错过广播——这就是为什么 DTIM Period 越大越省电、但广播延迟越高（VoIP、ARP 受影响）。

Listen Interval：主动声明睡多久

除了被动跟 DTIM，STA 还能在 Association Request 里声明 Listen Interval = N，意思是"我每 N 个 Beacon 醒一次"。N=1 每次都醒；N=10 每 10 个 Beacon（约 1 秒）醒一次，睡得更久。代价：AP 要为它缓存更多帧，缓存溢出就丢帧；下行延迟也变高。AP 据此分配缓存缓冲区，所以这是个关联时协商的参数——又回到第 7 课的 Association 流程。

More Data bit：AP 发缓存的下行帧时，如果 STA 还有更多缓存帧待取，置 More Data=1。STA 看到就继续醒着发 PS-Poll 取完；More Data=0 就回去睡。这是省电协议里"取件进度条"。

② U-APSD：让 AP"送货上门"

Legacy PS-Poll 的问题：每次取一个帧要一次 PS-Poll 往返，对 VoIP 这种双向小包业务很啰嗦——手机每 20ms 收一个语音包就醒一次发 PS-Poll，省电收益被控制帧开销吃掉。802.11e (WMM) 引入了 U-APSD (Unscheduled Automatic Power Save Delivery, 非调度自动省电投递)：

U-APSD 的核心改动

STA 不用发 PS-Poll "拉取"缓存帧，而是用上行数据帧（或 QoS Null 帧）当触发，AP 收到触发帧后主动把缓存的下行帧"推"过来。

效果：一次唤醒里，STA 发上行语音包 → 同一次唤醒里就收到下行语音包，双向一次唤醒搞定。对 VoIP 这类周期双向业务，省电效果远超 PS-Poll。

U-APSD 按 AC（接入类别，第 5 课 EDCA 的四个 AC）配置——可以只让语音 AC 走 U-APSD，尽力而为 AC 走 legacy PS-Poll。这种"按业务类型选省电策略"的精细度，是 WMM-PS 比 legacy 强的地方。Wi-Fi Alliance 的 WMM-PS 认证就是基于 U-APSD。

③ TWT：Wi-Fi 6 的 IoT 杀器

Legacy PS-Poll 和 U-APSD 都绑死 Beacon/DTIM 周期——STA 必须按 Beacon 节奏醒来，最多睡几百毫秒。对智能门锁、温湿度传感器这种"几十秒上报一次"的 IoT 设备， Beacon 周期（102.4 ms）依然太频繁，省电收益有限。

802.11ax (Wi-Fi 6) 从 802.11ah（IoT/_sub-GHz_ 专用 amendment，你可能有印象）借来了 TWT (Target Wake Time, 目标唤醒时间)：

TWT 的革命性

STA 和 AP 显式协商一个未来唤醒时刻表（Service Period），STA 只在约定的窗口醒来收发，完全脱离 Beacon/DTIM 周期的束缚。

效果：STA 可以睡几十秒甚至几分钟，醒来几十毫秒收发完再睡。这是 Wi-Fi 在 IoT 续航上第一次逼近 BLE/LoRaWAN 的水平。智能门锁、电池传感器、资产标签是典型受益场景。

TWT 协商通过 Action 帧完成（Individual TWT）或 Beacon/广播通告（Broadcast TWT）。STA 可以有多个 TWT 协议（不同业务用不同周期），AP 也能用它做调度——把不同 STA 的唤醒窗口错开，避免同时醒来抢占信道（这对密集 IoT 部署尤其有用）。

桥梁 / From sub-GHz

TWT 在精神上非常接近 LoRaWAN 的 Class B（定时 beacon 同步接收窗口）和你自己做 sub-GHz 协议栈时设计的"预约唤醒"。核心都是"协议层约定时间窗口 + 射频按需开关"。区别：Wi-Fi 的 TWT 协商在已关联的连接上进行（要先进 State 3），LoRaWAN 则是网络层 join 后通过 MAC 命令配置。你做过的"睡-醒"调度逻辑能直接迁移。

三代省电机制对比

机制引入取件触发睡眠粒度最适合效率
Legacy PS-Poll 802.11-1997 STA 发 PS-Poll 逐个取 Beacon/DTIM 周期 (~100ms) 偶发下行 基准
U-APSD 802.11e (WMM) 上行数据帧触发，AP 推送 仍绑 Beacon/DTIM VoIP 双向周期 较好
TWT 802.11ax (Wi-Fi 6) 协商的 Service Period 秒~分钟级，脱离 Beacon IoT（传感器/门锁） 最好

对驱动栈：省电机制落在代码哪里

对驱动重要 / For Driver Stack

省电是 Wi-Fi 驱动里最容易出 bug、最难调的部分之一。落点：

  • Power Management bit：每个发往 AP 的帧，Frame Control 里这一位要正确设置。驱动 tx 路径根据 STA 当前省电状态填这一位——填错 AP 不会缓存，下行丢帧。
  • Beacon 解析：rx 路径要解析 Beacon 的 TIM 元素，检查自己的 AID 位。SoftMAC 在 mac80211 的 ieee80211_rx_mgmt_beacon 里做；FullMAC 在固件里做，驱动只上报"有缓存"事件。
  • PS-Poll / 触发帧构造：检测到 TIM 里自己位=1，驱动要构造 PS-Poll（或 U-APSD 触发帧）发出去。AID 要填对（第 7 课关联时拿到的）。
  • 唤醒调度：硬件要能按 Listen Interval / TWT Service Period 定时唤醒射频。这通常涉及芯片的睡眠定时器、固件和驱动的电源状态机协调。睡眠后唤醒丢失 Beacon、唤醒太晚错过 DTIM，是常见 bug。
  • 调试难点：省电 bug 表现为"偶发丢包/延迟高/对端收不到"，抓包看到的是"PS-Poll 发了但 AP 没回"或"Beacon TIM 位和实际缓存不符"。要用 iw powertool / iw dev wlan0 get power_save 看当前策略，配合 monitor 抓包看帧交换。
  • 上层接口：用户态通过 nl80211 的 NL80211_CMD_SET_POWER_SAVE 切换省电开关。Android/iOS 还有厂商私有接口控制 TWT 参数。

自检：省电机制吃透了吗

这课的钉子：省电是"预约 + 取件"的协议设计

学完这课，你脑子里应该有 Wi-Fi 省电的统一骨架：

  1. 三步循环：声明睡眠 → 睡眠（AP 缓存）→ 醒来取件
  2. 三代演进：Legacy PS-Poll（按 Beacon）→ U-APSD（触发推送）→ TWT（独立时刻表）
  3. 三个伏笔兑现：AID（PS-Poll/TIM 索引）、TIM（Beacon 位图通知）、PS-Poll（取件帧）

你现在能做的：

  • 看到 Beacon 的 TIM/DTIM，知道哪些 STA 有缓存、广播什么时候发
  • 理解 PS-Poll 为什么用 AID 不用 NAV
  • 区分 Listen Interval（被动跟 Beacon）和 TWT（主动协商时刻表）
  • 把省电机制和你做 BLE/sub-GHz 的低功耗调度经验做类比
  • 定位省电类 bug 时知道看 Power Mgmt bit、TIM 解析、唤醒定时
